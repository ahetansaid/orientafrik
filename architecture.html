Étude de faisabilité · Note technique

Architecture MVP révisée — ORIENTAFRIK

Plateforme tripartite d'orientation post-bac. Cette note fige la stack, tranche la génération du Plan de Parcours, et recentre le périmètre pour tenir la fenêtre de lancement.

Bénin · Cotonou métropole Email-first (WhatsApp/SMS → Phase 2) Bootstrap · sans levée Drwintech SaaS Solutions

01 Verdict de faisabilité

Faisable — mais le risque n'est pas technique. La stack est saine. Les trois vraies menaces sur le MVP sont, dans l'ordre :

1. Le contenu — 8 Plans de Parcours vérifiés produits par une personne à temps partiel. C'est le chemin critique.  ·  2. La bande passante solo — un dev qui porte dev + design + marketing + compta. ·  3. Deux briques — génération PDF et split de paiement, tranchées ci-dessous.

02 Stack MVP figée

Mainstream, économique, cohérente avec l'existant Drwintech. Choix orientés « atteindre vite et bien » : chaque brique est managée, à plan gratuit généreux, sans ops lourde.

03 Plan de Parcours — l'approche tranchée

Un seul objet de données assemblé une fois, rendu de deux façons. L'infographie gratuite est du HTML (instantané, partageable). Le PDF payant passe par react-pdf, jamais par un navigateur headless.

GRATUIT · accroche de masse

Infographie inline « 2 pages »

  • React Server Component, HTML/Tailwind
  • Rendu instantané, léger sur 4G
  • Top 3 filières + score expliqué, écoles, bourses, calendrier
  • Partageable par capture / carte Open Graph
200 F · signal de valeur

PDF documenté 10-12 pages

  • @react-pdf/renderer dans un route handler Node
  • Curriculum détaillé, comparatif écoles, témoignages, sources
  • Généré à la volée puis caché dans Supabase Storage
  • Servi via URL signée après paiement (HTTP 402 sinon)
Profilage B1 → assemblerPlan() → objet PlanParcours → InfographieParcours (gratuit)  │  paiement 200 F → renderToBuffer(PlanParcoursPDF) → Storage → URL signée

Pourquoi pas Puppeteer. Le binaire Chromium (~300 Mo) dépasse la limite de bundle Vercel (250 Mo) et le serverless n'autorise pas de sous-processus. react-pdf élimine cette classe entière de problèmes.

04 Arbitrages de périmètre — le « MVP dans le MVP »

Les 9 briques ne sont pas toutes nécessaires au soft-launch. On coupe ~80h pour rendre la marge du planning au vert, sans toucher à la valeur perçue par le bachelier.

BriqueDécisionArbitrage
B1 Profiler · B2 Parcours · B3 PlanCŒUR La valeur bachelier. À soigner — c'est le différenciateur.
B4 Écoles · B5 BoursesCŒUR Catalogue + matching simple. Nécessaires au Plan.
B6 Réservation consultationRÉDUIRE Un seul consultant → créneaux fixes + paiement, pas un moteur de dispo complet.
B7 Dashboard ÉcoleREPORTER 5-8 écoles → rapport hebdo automatisé (email/PDF). Vrai dashboard = Phase 2. ~35h
B8 Dashboard ConsultantMINIMISER Le consultant est co-fondateur → il a l'admin B9. Profil public suffit. ~25h
B9 Admin + split paiementRÉDUIRE Fedapay sans split, reporting simple, réconciliation comptable. ~20h

05 Les 3 côtés & leur isolation (RLS)

Sécurité multi-tenant posée dès le schéma, pas après. Deny-by-default, puis ouverture au strict via des policies Postgres et des helpers security definer.

Bacheliers gratuit

Profilage, infographie, consultation N1 gratuite. Paient PDF (200 F) et consultations.

RLS : ne voit que ses profils, plans, paiements, réservations.

Écoles CPA B2B

Publiques gratuites au catalogue. Privées partenaires : commission à l'inscription effective.

RLS : is_ecole_member() → ne voit que les candidatures de son école.

Consultants 20 % / 80 %

En MVP : le co-fondateur seul. Profil public si vérifié, calendrier, revenus.

RLS : ne voit que ses créneaux et consultations.

06 À dérisquer avant de coder le produit

  • POC PDF (1-2 j) — valider react-pdf sur un parcours réel, polices locales, rendu Storage.
  • Fedapay — confirmer flux mobile money + webhook de confirmation (le split est Phase 2, hors sujet MVP).
  • Moteur de contenu — prototyper l'assemblage assisté IA sur un parcours avant d'en produire 8 à la main.
  • Arbitrage calendrier — confirmer que le build a démarré ; sinon assumer la fenêtre juillet-août 2027 plutôt qu'un janvier 2027 dégradé.
